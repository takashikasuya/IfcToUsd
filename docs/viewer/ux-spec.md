# UX・デザイン改良仕様（Epic E8）

Webビューワー（`ifc2usd/viewer/`）の視覚品質・操作性を段階的に引き上げるための仕様。
機能追加ではなく「既にある機能を、綺麗に・気持ちよく・迷わず使える」ようにすることが目的。
実装単位は `docs/viewer/backlog.md` の Epic E8 に対応する。

## 1. 背景と現状の課題

MVP〜E7で機能面（ツリー・選択・ボクセル・断面・SDFスライス・ワイヤフレーム）は揃ったが、
以下のUX上の弱点が残っている（ファイル参照は現行実装）:

| # | 課題 | 現状の実装 |
| --- | --- | --- |
| C1 | 選択ハイライトが弱い | メッシュはemissiveを`0x3355ff`にするだけ（`highlightMesh`）。周囲が暗い色の要素だと視認しづらく、輪郭が出ないため「どこからどこまでが選択要素か」が分かりにくい |
| C2 | 3D→ツリーの誘導が無い | 3Dクリックでツリー行に`.selected`は付くが、**スクロールも展開もされない**ため、大きいモデルでは選択行が画面外のまま |
| C3 | ツリーが常に全展開 | `renderTreeNode`は折りたたみ無しの入れ子UL。76要素のToyodaLabで既に縦に長く、1000要素超では実用性が落ちる |
| C4 | ホバーのフィードバックが無い | 3D側・ツリー側とも、クリックするまで「何が選ばれるか」の予告が無い（ツリーは行の背景色のみ） |
| C5 | 検索・絞り込みが無い | 「AW2という窓を見たい」に対してツリーを目視で探すしかない |
| C6 | プロパティパネルが素のdl | キーの整形・単位・コピー手段なし。GUIDの手動転記が必要 |
| C7 | ツールバーが横一列で飽和 | 表示モード/LOD/Wireframe/Section/SDFが1行に並び、既に窮屈。今後の機能追加（E9のツインモード等）を受け入れる余地が無い |
| C8 | デザイントークン不在 | 色・余白・フォントが`index.html`のインラインCSSに直書きで、統一感の担保が個人の注意力頼み |
| C9 | ボクセルがほぼ真っ黒 | Issue #39（`MeshStandardMaterial`+ライティングの問題）。ボクセル表示自体の視覚品質の土台が壊れている |

## 2. デザイン原則（この順で優先）

1. **ビルド不要を維持する。** ES modules + vendored three.js の現行構成を崩さない。
   フレームワーク（React等）・CSSプリプロセッサ・バンドラは導入しない。
2. **テスト可能性を維持する。** すべての視覚状態はPlaywrightで検証できること
   （DOM状態 + `?e2e`での画素サンプリング）。`window.ifc2usdViewer`のフックは最小限で拡張する。
3. **性能予算を守る。** NFR-2（50万ボクセルインスタンス）を悪化させない。
   ハイライト追加は+2 draw call以内、ホバーのレイキャストは1フレーム1回以内、
   毎フレームのアロケーションを増やさない。
4. **既存の設計規約に従う。** レイキャスト対象は`.visible`に頼らず明示リストで組む
   （CLAUDE.md記載のRaycaster規約）、可視状態の決定は`applyDisplayState()`へ一元化、
   断面クリップ（`renderer.clippingPlanes`はグローバル）との整合を常に確認する。

## 3. 機能仕様

### 3.1 選択ハイライトの強化（E8-1）

**目標**: 選択要素が「輪郭付きで浮かび上がり」、それ以外が「引いて」見える。

- **輪郭（アウトライン）**: 選択メッシュの複製を法線方向にわずかに膨らませ、
  `MeshBasicMaterial({ color: ハイライト色, side: THREE.BackSide })`で本体の背後に描く
  「バックフェイス・ハル方式」を採用する。
  - 採用理由: EffectComposer + OutlinePass（ポストプロセス方式）は品質は高いが、
    (a) vendoredアドオンの追加（EffectComposer/ShaderPass/OutlinePass等）、
    (b) フルスクリーンレンダーターゲットのコスト（E2E環境のSwiftShaderで顕著）、
    (c) 全体のレンダリングパイプライン変更、を伴う。バックフェイス方式は
    +1 draw call・既存パイプライン無変更で足り、この用途（単一選択の強調）には十分。
  - ボクセル側: 選択要素のインスタンスだけを含む一時的な`InstancedMesh`
    （単位Boxを~1.06倍スケール、BackSide）を同様に生成する。E7-3の
    `instanceIndicesByGuid`がそのまま流用できる。
  - 断面クリップとの整合: アウトラインメッシュにもグローバルクリップが効くため
    追加対応は不要（確認テストは必要）。
- **ゴースト表示（他要素を引かせる）**: 選択中、非選択要素を半透明
  （`opacity≈0.15, transparent, depthWrite:false`）にするトグル（既定off）。
  - **共有マテリアルの罠（要検証・要対応）**: GLTFLoaderはglTFのマテリアルindexを
    共有するため、`material.opacity`の直接変更は同色の他要素へ波及しうる。
    既存の`highlightMesh`（emissive変更）にも同じ潜在問題があるため、E8-1の実装時に
    「同色要素を複数選択した場合にemissiveが波及しないか」をまず検証し、
    波及するなら**初回変更時にマテリアルをclone**する戦略
    （`mesh.userData.__ownMaterial`）へ統一する。ゴーストは共有の
    「ゴースト用マテリアル1個への差し替え+復元」で実装し、cloneの大量発生を避ける。
- **選択へのフィット**: ツリー行/3Dのダブルクリックで
  `fitCameraToBox(getBoundingBoxOfGuid(guid))`。既存関数の呼び出しのみ。

### 3.2 ホバー連携（E8-2）

- 3D側: `pointermove`でレイキャスト（`currentRaycastTargets()`を流用、rAFで
  スロットリング、ドラッグ中はスキップ）し、ヒット要素に「選択より弱い」ホバー表現
  （emissiveを`0x222a44`程度）を付ける。カーソルを`pointer`にする。
- ツリー側: 行の`mouseenter/leave`で同じホバー表現を3Dへ適用（双方向）。
  3Dホバー中はツリー行に`.hovered`クラス（背景をわずかに明るく）。
  ホバーでは**スクロールしない**（視線が飛んで酔うため。スクロールは選択時のみ）。
- ホバーと選択は独立に管理する（選択中要素にホバーが重なっても選択表現を優先）。

### 3.3 ツリービューの改良（E8-3）

- **折りたたみ**: 各中間ノードに開閉トグル（▸/▾）。初期状態は
  「Storey（IfcBuildingStorey）まで展開、要素は畳む」。開閉状態はメモリ内のみ
  （ページ再読込で初期化、永続化しない）。
- **選択行の自動展開+スクロール**: `selectByGuid`時に祖先を展開し
  `scrollIntoView({ block: "nearest" })`。C2の直接解消。
- **検索/絞り込み**: ツリー上部にテキスト入力。名前/クラス/GUIDの部分一致
  （大文字小文字無視、150msデバウンス）。マッチ行とその祖先だけを表示し、
  マッチ部分をハイライト。クリアで全表示へ戻る。
- **クラス色チップ**: 各行の先頭に要素色の小さな●を表示する。色情報は現在
  `scene.json`に無いため、`scene_index.py`のノードへ`color`（diffuseColor）を追加する
  （スキーマ後方互換: 無ければチップ非表示）。
- **行アクション**: ホバー時に行末へ「このみ表示（isolate）」「フィット」の
  小アイコンを出す。isolateは対象サブツリー以外を非表示にし、もう一度押すと解除。

### 3.4 プロパティパネルの改良（E8-4）

- キーを定義順で整形表示（GUID/class/Name/LongName/Description/Latitude/Longitude）。
  クラスはチップ（3.3と同じ色）+ `Ifc`プレフィックスを落とした表示名。
- GUID行にコピーボタン（`navigator.clipboard`、失敗時はテキスト選択フォールバック）。
- 緯度経度は度分秒→10進変換済みの値に単位を付けて表示。
- 未選択時は空白ではなく操作ガイド（「要素をクリックまたはツリーから選択」）を表示。
- （E9連携の予約）パネル下部に「Live Data」セクションの挿入位置を確保しておく。

### 3.5 ツールバー・レイアウト・テーマ（E8-5）

- **ツールバーのグループ化**: 「表示（モード/LOD/Wireframe）」「解析（Section/SDF）」を
  区切り線＋小見出しでグループ化。将来グループ（E9「Live」）の追加余地を持たせる。
  各コントロールに`title`ツールチップ。
- **デザイントークン**: `:root`のCSSカスタムプロパティへ集約する。
  最低限: `--bg`, `--bg-panel`, `--border`, `--text`, `--text-dim`, `--accent`
  （選択・ハイライトの`#3355ff`系）, `--radius`, `--space-1..3`。
  viewer.js内のハードコード色（背景`0x202020`、ハイライト`0x3355ff`）は
  CSS変数から読み取る形へ寄せる（`getComputedStyle`で初期化時に1回解決）。
- **パネルの開閉**: 左（ツリー）・右（プロパティ）パネルを折りたたみ可能にする
  （狭い画面対応）。開閉で`#viewport`のサイズが変わるのは既存のResizeObserverが吸収する。
- **キーボードショートカット**: `F`=選択（無ければ全体）へフィット、`Esc`=選択解除、
  `W`=ワイヤフレーム、`1/2/3`=表示モード。input要素フォーカス中は無効。
  ショートカット一覧を`?`で表示（簡易オーバーレイ）。

### 3.6 ボクセル描画品質の修復（E8-6 = Issue #39）

ボクセルがライティング下でほぼ真っ黒になる既存バグの修正。E8-1のボクセル側
ハイライトやE9のボクセルヒートマップは「ボクセルの色が見える」ことが前提のため、
**Epic E8の中で最初に着手する**。調査方針はIssue #39に記載済み
（InstancedMesh + `vertexColors` + `MeshStandardMaterial`とライトの相互作用、
`_voxelUnitBox`の法線、SwiftShader固有かの切り分け）。

## 4. 受け入れの共通基準

- 各ストーリーはPlaywright E2Eテストを伴う（DOM状態 + 必要に応じ`?e2e`画素検証）。
- 既存170+テストを破壊しない。とくに`currentRaycastTargets`・表示モード・
  ハイライト復元（E7-3）の回帰に注意する。
- 実データ（`files/ToyodaLab.ifc`）でのスクリーンショット確認を実施する。

## 5. スコープ外（このEpicではやらない）

- ツリーの仮想化（数万要素対応）。必要になった時点で別ストーリー化する。
- ライト/ダークテーマ切替（トークン化はするが、テーマは現行ダークのみ）。
- モバイル最適化（タッチ操作の基本はOrbitControlsが既に提供している範囲まで）。
- ポストプロセスによる高品位アウトライン（バックフェイス方式で不足と判明した場合に再検討）。
