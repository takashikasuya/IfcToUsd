IFC2USD

IFC（建築BIM）モデルを構造化された OpenUSD ステージへ変換し、ボクセル解析・glTFエクスポート・ 自己完結型Webビューワー・ビルOS連携のデジタルツイン表示までを、ひとつのCLIで提供するツールキット。

Python ≥ 3.10 OpenUSD (usd-core) ifcopenshell 0.8 three.js（同梱・CDN不要） uvのみでセットアップ完結
目次 概要 アーキテクチャ 機能 使い方 デジタルツイン表示 利用技術 テストと品質

概要

IFC2USD は、建築BIMの標準交換フォーマットである IFC のモデルを、 ピクサー発のシーン記述フォーマット OpenUSD の構造化ステージへ変換する。 IFCの空間階層（Site → Building → Storey → Space / Element → Object）を USD の Xform 階層として そのまま再現し、各エレメントにメッシュ・UsdPreviewSurface マテリアル・IFC由来のメタデータ （GUID・IFCクラス・名称・緯度経度など）を customData として付与する。

本システムの役割は大きく2つに分かれる。

1. IFC→USD変換（正本の生成）

IFCの階層・ジオメトリ・マテリアル・メタデータを忠実にUSDへ変換する。 この変換結果が以降のすべての処理の唯一の正本（single source of truth）となり、 後段の処理が書き換えることはない。

2. 変換結果の統合閲覧

正本USDからボクセル・SDF（符号付き距離場）・glTF・シーンインデックスなどの 派生アセットを生成し、USDネイティブビューワーとブラウザの2系統の閲覧経路を提供する。 さらにビルOSと連携すれば、実センサー値のデジタルツイン表示までできる。

設計思想: 閲覧手段を1つの独自ビューワーに寄せるのではなく、 Hydra（USDのレンダリング抽象化層）の考え方に倣って「シーンデータの用意」と「実際のレンダリング」を分離している。 USDを直接読める既存ツール（usdview / Blender / Omniverse）には独自レンダラを書かずにデータ側で対応し、 ブラウザにはUSDをパースさせず軽量な派生ファイルだけを配る。

アーキテクチャ

設計上の不変条件

不変条件内容
正本は1つ convert が生成するUSDが唯一の正本。voxelize / export-gltf / serve は読むだけで書き換えない。 PointInstancerボクセルレイヤーも既存USDへの「参照」を持つ独立ファイル。
派生アセットは付加的 voxels.json / sdf.json / twin.json / space_voxels.json はどれも「あれば機能が増え、 無ければ既存動作を一切変えない」。存在は scene.json の assets.* キーで伝える。
ブラウザはUSDを読まない WebビューワーはGLB・scene.json ほか軽量な派生ファイルだけを読む。 USDのパースはすべてPython側で完結する。
認証情報はサーバー側のみ ビルOSのURL・認証情報は twin-config.json（サーバー側）にのみ存在。 ブラウザは同一オリジンのホワイトリスト済みプロキシ （/api/twin/values・/api/twin/history）越しにしか上流へ到達できず、 エラー応答にも上流URLは含まれない。

機能

IFC→USD変換

空間階層をXformツリーで忠実に再現。要素ごとにメッシュ+UsdPreviewSurfaceマテリアル+ GUID/クラス/名称/緯度経度のcustomData。Z-UP既定、--y-up切替。

ボクセル化（LOD付き）

表面ボクセルまたはflood-fillによる内部充填。複数--sizeでLOD化し、 JSON（Morton符号化+delta+RLE圧縮）とPointInstancer USDレイヤーの2形式で出力。

glTF(GLB)エクスポート

Webビューワーの内部形式であり単体でも利用可能。metallic/roughnessを正しく引き継ぎ、 IFC要素との対応をノードextrasに保持。

自己完結Webビューワー

階層ツリー・検索・アイソレート、クリック選択+プロパティパネル、メッシュ/ボクセル表示切替、 LOD切替、断面クリップ、ホバー/アウトラインハイライト。オフライン環境でも動作。

SDF断面オーバーレイ

選択要素のnarrow-band SDF水平断面を半透明表示（白=表面、青=内部、橙=外部）。 余分な計算コストがあるためopt-in（--sdf-slices）。

デジタルツイン表示

Live色マッピング（turboカラーマップ+stale値の彩度低下）、部屋単位ボクセルヒートマップ、 時系列再生（Playback）。ビルOSのセンサー値を3Dモデル上へ可視化。

使い方

セットアップ

uv だけで完結する。usd-core と ifcopenshell が PyPI配布されているため、condaやUSDのソースビルドは不要。 

uv sync          # .venv を自動作成し、dev グループ（pytest / Playwright）まで導入

基本フロー

# 1. IFC→USD変換（既定で output/<name>_structured.usda）
uv run ifc2usd convert files/ToyodaLab.ifc

# 2. ブラウザで閲覧（http://127.0.0.1:8000 が開く）
uv run ifc2usd serve output/ToyodaLab_structured.usda

コマンド一覧

コマンド説明主なオプション
convert <ifc> IFC→USD変換（サブコマンド省略の ifc2usd <ifc> でも可） -o 出力先 / --y-up / -v
voxelize <usda|ifc> 占有ボクセル化。JSON + PointInstancer USDレイヤーを出力 --size（複数指定でLOD）/ --fill 内部充填 / -o
export-gltf <usda> USD→GLBエクスポート -o 出力先
serve <usda> Webビューワー配信（127.0.0.1のみ） --port / --no-open / --sdf-slices / --twin / --space-voxels
space-voxelize <ifc> IfcSpace（部屋）のボクセル帰属マップ生成（ヒートマップの前提データ） --reference 変換済みUSD（必須・原点合わせ）/ --size / -o

USDネイティブツールでの閲覧（経路A）

uv run ifc2usd voxelize output/model.usda --size 1.0 --size 0.5
# → 出力された PointInstancer .usda を usdview / Blender / Omniverse でそのまま開く
#   voxelLOD variantSet で --size ごとのLODを切替できる

デジタルツイン表示モード

GUTPビルOSのREST APIと連携し、温度・CO2などの実センサー値を3Dモデル上へ可視化する。 ビルOSのデータモデルはIFC GUIDを持たないため、mapping.json が GUID↔計測ポイントの結合層になる（手動記述 / IFCプロパティからの自動生成 / ビルOS customTags逆引きの3経路）。 

# 部屋単位ヒートマップ用の帰属マップを事前生成
uv run ifc2usd space-voxelize files/ToyodaLab.ifc \
    --reference output/ToyodaLab_structured.usda --size 0.5 -o output/spaces.json

# デジタルツインモードで起動
uv run ifc2usd serve output/ToyodaLab_structured.usda \
    --twin twin-config.json --space-voxels output/spaces.json
機能内容
Live メトリックを選ぶと /api/twin/values をポーリングし、バインド済み要素を turboカラーマップで着色+凡例表示。ポーリング間隔×3より古い値は彩度を落とし、 「止まっているのに動いて見える」事故を防ぐ。選択要素のプロパティパネルには 最新値+スパークラインのLive Dataセクションが出る。
空間ヒートマップ 部屋（IfcSpace）単位の集計値（平均）でボクセルレイヤーを着色。 IfcSpaceジオメトリの無いモデルではStorey配下の要素メッシュ着色へフォールバック。
Playback（時系列再生） /api/twin/history を対象ポイントぶん一括取得してフレーム列へ整形し、 スライダー/自動再生でスクラブ。再生中の逐次fetchはしない。 色計算はLive・ヒートマップと完全に同一の関数を通る（表示経路を分岐させない）。
セキュリティ ビルOSのURL・認証情報はサーバー側の設定ファイルにのみ存在。ブラウザへ配る twin.json はメトリック定義とバインディングのみ。プロキシは values/history の2エンドポイントだけをホワイトリストし、制御系APIへは一切到達できない。

利用技術

技術用途
OpenUSD（usd-core ≥ 25.5） 変換先のシーン記述。Xform階層・Mesh・UsdPreviewSurface・variantSet・customDataを使用
ifcopenshell ≥ 0.8 IFCのパース・ジオメトリテッセレーション・プロパティ抽出（0.8の新API: 文字列キー設定、列優先行列など）

three.js（同梱） Webビューワーの描画。InstancedMeshによるボクセル描画、Raycasterによるクリック選択。 CDN参照なしでオフライン動作
Python標準ライブラリ http.server ローカル配信+同一オリジンtwinプロキシ。外部Webフレームワーク非依存
pytest / Playwright 変換〜ビューワーまでのE2Eテスト。実ブラウザでレンダリング画素をサンプリングして検証
アルゴリズム Morton(Z-order)符号化（63bit、自前実装）、delta+RLE圧縮、ボクセルグリッドflood-fillによる 内部判定、narrow-band SDF、turboカラーマップ多項式近似（Apache-2.0）

テストと品質

小さな合成IFCフィクスチャ（Site/Building/Storeyに色付きの壁2枚）を実際に変換し、 座標系・階層・ジオメトリ・マテリアル・ボクセル化・glTF・ビューワー・デジタルツイン連携までを 検証するテストスイート（350+件）を同梱している。外部データ不要で動く。

  • Webビューワー関連はPlaywrightの実ブラウザE2E。データの正しさだけでなく、 実際にレンダリングされた画素の色をサンプリングして「画面に正しく届いているか」まで確認する （glTFのmetallic/roughness欠落やボクセルが黒く塗り潰されるといった、データ検査では 捕まらない実バグをこの方式で検出してきた）
  • ビルOSはモックHTTPサーバーで代替し、ネットワーク接続なしでデジタルツイン機能をテストできる
uv run pytest                              # 全テストの実行
uv run python tests/generate_fixture.py    # テスト用IFCフィクスチャの再生成
